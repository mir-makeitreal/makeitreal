## 1. Resolved Disagreements

### 1A. Graph Visualization: React Flow vs d3-force vs Something Else

**Decision: Two-tier approach — Mermaid pre-rendered SVG for static HTML + d3-force for live server.**

**Rationale:** The project has TWO dashboard modes:
1. Static HTML (`file:///`) — generated by the engine, works offline
2. Live server (`http://localhost`) — Change 10, enables interactivity

For static HTML: Mermaid pre-rendered SVG with vanilla JS pan/zoom (~50 lines).
For live server: d3-force (66KB bundled) with force-directed layout, d3-zoom for
pan/zoom, click-to-inspect, and cross-panel linking. This is committed as a pre-built
bundle in `src/server/public/bundle.js` — no runtime build step.

**Why d3-force over React Flow:**
- d3-force+zoom+select+drag = 66KB raw vs React Flow+React = 194KB raw (measured with esbuild)
- Only 2 graph panels (topology, DAG); rest are DOM (contracts, file tree, sequences)
- No framework dependency for what is fundamentally a view-only dashboard
- Cross-panel linking (click node → highlight in file tree, contracts) is a store pattern,
  not a framework feature

**Why not pre-rendered SVG only:**
- The live server exists (Change 10) — there's no reason to limit interactivity when
  we have localhost. Force-directed layout responds to data changes in real-time.
- d3-force enables: click-to-inspect detail panel, animated node grouping, live updates
  as sub-agents complete tasks (nodes turn green in real-time)

### 1B. Database vs preview-model.json

**Decision: preview-model.json IS the database. No SQLite, no separate store.**

**Rationale:** The system generates artifacts as JSON files in a run directory. These
files are the source of truth. preview-model.json is a computed view assembled from
them. Adding a database layer creates sync problems, migration burden, and violates
the "generated artifact" philosophy. The live server (Change 10) reads preview-model.json
and watches for changes via fs.watch. This is sufficient for single-user localhost use.

### 1C. Recursive Sub-Agent NEEDS_DECOMPOSE Protocol

**Decision: Structured protocol with explicit contract.**

When a sub-agent determines its work item is too complex for single-pass implementation:

1. Sub-agent returns `{ status: "NEEDS_DECOMPOSE", reason: string, suggestedSplit: WorkItem[] }`
2. Orchestrator validates: current depth < max_depth (2)
3. If allowed: orchestrator calls plan-generator to create child work items with
   `parentWorkItemId` set, adds them to board, spawns child sub-agents
4. If at max depth: orchestrator forces the sub-agent to attempt implementation anyway,
   with a note that decomposition was denied
5. Parent work item moves to `blocked-by-children` lane until all children complete
6. On all children complete: parent auto-transitions to `verify` lane

**Max depth:** 2 (original → child → grandchild). Configurable via project config
but defaults to 2. Depth 3+ is explicitly blocked to prevent runaway recursion.

**Board integration:** Child work items get IDs like `{parentId}:sub:{index}`.
They appear in the same board with `parentWorkItemId` field linking them.

### 1D. Browser Approval vs Terminal Approval

**Decision: Browser approval SUPPLEMENTS terminal approval. Terminal remains primary.**

**Rationale:**
- Terminal approval already works (interactive-approval.mjs, 386 lines, well-tested)
- Claude Code's hook system is terminal-native; browser approval cannot trigger hooks
- Browser approval is a convenience for visual review, not a replacement

**Implementation:**
- The live localhost dashboard (Change 10) shows an "Approve / Request Changes" UI
- Browser clicks send a POST to the localhost server
- The server calls the same `decideBlueprintReview()` / `recordBlueprintRevisionRequest()`
  functions that terminal approval uses
- The WebSocket pushes the updated status back to all connected browsers
- Terminal still works exactly as before; browser is an additional surface

**Flow:**
```
Browser UI → POST /api/blueprint/review → decideBlueprintReview() → fs write
                                        → WebSocket broadcast status update
Terminal   → hook → applyNativeBlueprintReviewDecision() → decideBlueprintReview() → fs write
```

Both paths converge on the same review.mjs functions. No duplication.

## 5. Preview-Model Schema Enhancements

Current schema version: "1.0". New version: "1.1" (backward compatible).

### New Fields (additive only — no existing field removal or rename)

```jsonc
{
  "schemaVersion": "1.1",                    // bumped from "1.0"
  "generatedAt": "...",

  "run": { /* unchanged */ },

  "blueprint": {
    /* all existing fields unchanged */

    // NEW: metrics summary for hero section
    "metrics": {
      "moduleCount": 5,                      // count of architecture.nodes
      "contractCount": 3,                    // count of contracts
      "workItemCount": 8,                    // count of work items
      "gatesPassed": 2,                      // count of gates with status "passed"
      "gatesTotal": 4,                       // total gate count
      "estimatedEffortDays": 12              // sum of work item estimates (if available)
    },

    // NEW: recursive decomposition tracking
    "decompositionDepth": 0,                 // 0 = original, 1 = first decompose, 2 = max
    "parentBlueprintId": null                // null for top-level, ID for children
  },

  "status": {
    /* all existing fields unchanged */

    // NEW: server connectivity
    "serverPort": null,                      // populated when live server is running
    "lastWebSocketPush": null                // ISO timestamp of last push
  },

  "board": {
    /* all existing fields unchanged */

    // NEW: work item recursive fields
    "lanes": [{
      "name": "...",
      "workItems": [{
        /* all existing fields unchanged */
        "parentWorkItemId": null,            // NEW: null for top-level, parent ID for children
        "childWorkItemIds": [],              // NEW: IDs of decomposed children
        "decompositionDepth": 0              // NEW: nesting level
      }]
    }]
  },

  // NEW: top-level section for live server state (only populated when server runs)
  "server": null
  // When running:
  // "server": {
  //   "port": 3847,
  //   "startedAt": "...",
  //   "connectedClients": 2
  // }
}
```

### Backward Compatibility Rules

1. All new fields default to null or empty — consumers that don't know about them skip safely
2. schemaVersion "1.0" consumers ignore unknown fields (JSON is forward-compatible)
3. The dashboard renderer handles both "1.0" and "1.1" models
4. preview-model.mjs computes `metrics` from existing data (no new inputs needed)

### Snapshot Testing for Dashboard Sections

Each section renderer test:
1. Loads the canonical fixture from examples/canonical/.makeitreal/runs/feature-auth/
2. Builds preview model via buildPreviewModel()
3. Calls the section renderer with the model
4. Asserts: output is a non-empty string containing expected HTML markers
