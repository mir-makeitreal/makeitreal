# Make It Real — Architecture Change Plan

## Status: DRAFT — Awaiting Multi-Model Review

## 1. Problem Statement

Make It Real is a Claude Code plugin that implements "Blueprint First" development:
broad goal → architecture/contracts → parallel sub-agent implementation → gated verification.

**What works well (DO NOT TOUCH):**
- Kanban state machine + Gate system (15 lanes, gate-guarded transitions)
- 3 Hook enforcement (pre-tool-use, user-prompt-submit, stop)
- Error model `{code, reason, evidence[], recoverable}`
- Blueprint fingerprinting (SHA-256 staleness detection)
- Orchestrator Native Claude Code Task path (start → finish → complete)
- Completion Policies (implementation/domain-pm/integration-evidence)
- 272 passing tests covering the full pipeline

**What needs to change:**
1. Dashboard renders 4018 lines of inline HTML — unclear information hierarchy
2. plan-generator has hardcoded domain knowledge (book/order/DataTable regex)
3. No user-facing documentation (README, getting-started, architecture docs)
4. preview-model.json duplicates data across 5+ sections (955 lines canonical)
5. Duplicate source tree (root `src/` is exact copy of `plugins/.../src/`)

## 2. Design Principles

- **Minimal invasion**: Keep all 272 tests passing. Change rendering, not the pipeline.
- **Existing artifacts stay**: prd.json, design-pack.json, work-item-dag.json, board.json, contracts/ — these schemas are the foundation. Don't redesign them.
- **Separate concerns**: Data (engine) → ViewModel (dossier) → View (new renderer)
- **SDK-grade docs**: A developer should understand the system in 5 minutes.

## 3. Changes Overview

### Change A: Replace Dashboard Renderer

**Current:** `src/preview/render-dashboard-html.mjs` (4018 lines)
Generates full HTML+CSS+JS SPA as template literals. Single 12000px scroll document.

**Proposed:** Replace with a standalone docs-site that consumes `preview-model.json`.

Architecture:
```
preview-model.json (existing, generated by engine)
       │
       ▼
docs-site/          (NEW - standalone React app)
├── package.json
├── src/
│   ├── App.tsx
│   ├── components/
│   │   ├── SystemPlacement.tsx    ← React Flow graph of modules + contracts
│   │   ├── TaskDAG.tsx            ← React Flow DAG of work items + dependencies
│   │   ├── WorkerTopology.tsx     ← Sub-agent assignment cards
│   │   ├── ResponsibilityMap.tsx  ← Ownership boundaries with file trees
│   │   ├── ContractSurfaces.tsx   ← API/module interface signatures
│   │   ├── ScenarioIndex.tsx      ← Sequence diagrams (React Flow or Mermaid)
│   │   ├── ApprovalScope.tsx      ← Fingerprint, paths, required work items
│   │   └── EvidencePanel.tsx      ← Verification/review status
│   ├── layouts/
│   │   ├── Sidebar.tsx            ← Sticky nav with section links + search
│   │   └── PageLayout.tsx         ← Responsive container
│   └── lib/
│       ├── parsePreviewModel.ts   ← Typed parser for preview-model.json
│       └── theme.ts               ← Design tokens (colors, spacing, type)
├── public/
│   └── index.html
└── vite.config.ts
```

**Key decisions:**
- React + Vite for fast build. Output is still a single `index.html` with inlined assets.
- React Flow for interactive architecture/DAG visualization
- The engine's `renderDesignPreview()` builds the model, then invokes `vite build` to produce `preview/index.html`
- `render-dashboard-html.mjs` gets replaced by a thin wrapper that runs the build
- Existing test contract: `renderDesignPreview()` returns `{ok, previewDir, errors}` — unchanged

**Why not Docusaurus/Nextra:**
- The dashboard is per-run, generated from each blueprint's data
- It's not a static docs site — it's a dynamic data visualization
- Must work as `file:///` URL (offline, no server)

**Rendering contract:**
```
INPUT:  preview-model.json (generated by preview-model.mjs — UNCHANGED)
OUTPUT: preview/index.html (self-contained, works as file:/// URL)
```

### Change B: Clean Up plan-generator Domain Hardcodes

**Current:** `plan-generator.mjs` lines 311-597 contain hardcoded domain knowledge:
- `fieldSchema()`: regex-based type inference ("isEnabled" → boolean)
- `sampleValueForSchema()`: "address" → `{line1: "1 Example St"}`
- `resourcePathFromRequest()`: lookup table of 11 REST resources
- `componentProfileFromRequest()`: DataTable-specific prop definitions
- `knownFields`: hardcoded list of 15 field names

**Proposed:** Extract domain inference into a pluggable config layer.

```
src/plan/domain-profiles/
├── index.mjs              ← Registry: loadDomainProfile(request, projectRoot)
├── api-profile.mjs        ← API type detection + scaffold (KEEP most of current logic)
├── component-profile.mjs  ← Component type detection + scaffold (KEEP)
├── module-profile.mjs     ← Module type detection + scaffold (KEEP)
└── defaults.mjs           ← Default field schemas, sample values (MOVE hardcodes here)
```

**What changes:**
- Move hardcoded lookup tables to `defaults.mjs` (explicit, discoverable)
- `loadDomainProfile()` checks for `.makeitreal/domain-profile.json` in project root
  - If present: use project-specific field schemas, resources, component props
  - If absent: fall back to current defaults
- plan-generator main flow (generatePlanRun) stays UNCHANGED
- All 272 tests keep passing — defaults.mjs produces exact same output

**What does NOT change:**
- The deterministic artifact generation approach (this is correct)
- The 3-way type classification (API/Component/Module)
- The responsibility decomposer
- The Gate validation of generated artifacts

### Change C: Eliminate Source Duplication

**Current:** `src/` and `plugins/makeitreal/dev-harness/src/` are identical copies.
`bin/harness.mjs` imports from root `src/`. Plugin sync script copies root → plugins.

**Proposed:** Make `plugins/makeitreal/dev-harness/src/` a symlink to root `src/`.

```bash
# In plugin directory:
plugins/makeitreal/dev-harness/src → ../../../src  (symlink)
```

**Risk:** Claude Code plugin resolution may not follow symlinks.
**Mitigation:** Test with actual Claude Code plugin install. If symlinks fail,
keep the copy but add a CI check that they're identical (current `plugin:sync --check`).

### Change D: Reduce preview-model.json Duplication

**Current:** 955 lines. Same contract data appears in:
- `blueprint.contracts`
- `blueprint.moduleInterfaces[].publicSurfaces[].contractIds`
- `blueprint.systemDossier.contractMatrix`
- `blueprint.systemDossier.contractSurfaces`
- `blueprint.systemDossier.modules[].publicSurfaces`

**Proposed:** Restructure preview-model.mjs to use references instead of copies.

```json
{
  "contracts": { "contract.auth.login": { ...full definition... } },
  "modules": [
    {
      "id": "ru.frontend",
      "publicSurfaces": [{ "contractRef": "contract.auth.login" }]
    }
  ],
  "dag": { "nodes": [...], "edges": [...] },
  "dossier": { "dependencyEdges": [...], "fileTree": {...} }
}
```

**Risk:** Breaking existing consumers of preview-model.json.
**Mitigation:**
- Check if anything besides render-dashboard-html.mjs reads preview-model.json
- If only the dashboard reads it, we control both sides
- Keep `preview-meta.json` unchanged (it's the public summary)

## 4. Implementation Order

```
Phase 1: Change E (docs) — Lowest risk, highest immediate value
Phase 2: Change B (plan-generator cleanup) — Refactor, not rewrite
Phase 3: Change D (preview-model dedup) — Prerequisite for new dashboard
Phase 4: Change A (dashboard replacement) — Biggest change, most visible

```

## 7. Open Questions for Reviewers

1. **React Flow vs Mermaid vs D3:** React Flow gives interaction but adds bundle size.
   Mermaid is lighter but less interactive. D3 is flexible but more code.
   The dashboard must work as `file:///` — does React Flow work without a server?

2. **Vite build integration:** The engine currently writes HTML directly. Adding a
   build step means node_modules must exist. Is this acceptable for a Claude Code
   plugin, or should we use a pre-built bundle?

3. **preview-model.json schema stability:** If we change the preview-model shape
   (Change D), should we version it? Or is it purely internal?

4. **Domain profile extensibility (Change B):** Should `.makeitreal/domain-profile.json`
   be a first-class feature, or just internal cleanup? If first-class, it needs
   schema validation and documentation.

5. **Docs site vs per-run dashboard:** Should there ALSO be a static documentation
   site (architecture overview, getting started) separate from the per-run dashboard?
   Currently only per-run dashboards exist.
